Part 1: Data Flow Between Screens

Making Screens Communicate

The navigation Prop's Other Half: The route Prop

In Session 7, we learned that every screen gets a navigation prop to perform actions. Now, we introduce its partner: the route prop.

navigation Prop

Purpose: To DO something (e.g., go to a new screen).

Key Method: navigation.navigate()

route Prop

Purpose: To GET something (e.g., data passed to this screen).

Key Property: route.params

Passing Data with navigation.navigate

We already know how to use navigate to switch screens. To pass data, we simply add a second argument: an object containing the data we want to send.

// The second argument is our "data package"
navigation.navigate('RouteName', { param1: value1, param2: value2 });

This is the fundamental mechanism for sending information forward in a stack.

Code: Sending the Chosen Number

Let's update the StartGameScreen. When the user confirms their number, we'll send it to the Game screen.

// In screens/StartGameScreen.tsx

function confirmInputHandler() {

Now, let's use the route prop in our GameScreen to catch the data.

// Screen components get navigation AND route props automatically
function GameScreen({ route, navigation }) {
  // We use the 'route' prop to access the parameters
  // and extract the userNumber we passed from the StartGame screen.
  const { userNumber } = route.params;

Best Practice: Typing Your Routes

Why is this important? Without types, TypeScript doesn't know what route.params might contain. You could misspell userNumber and your code would crash at runtime without any prior warning from the editor.

The solution is to create a central type definition that tells TypeScript the "shape" of our navigation stack and the parameters each screen expects.

You can learn more from the official docs: React Navigation: Type checking with TypeScript

Code: Typing the RootStackParamList

We'll define a type for our navigation stack. This is often done in App.tsx or a dedicated navigation/types.ts file.

import { NativeStackScreenProps } from '@react-navigation/native-stack';

// Define all screens and the params they expect
export type RootStackParamList = {
  StartGame: undefined; // StartGame expects no params
  Game: { userNumber: number }; // Game expects 'userNumber'
  GameOver: { rounds: number }; // GameOver will expect 'rounds'
};

// Define the prop types for a specific screen
export type GameScreenProps = NativeStackScreenProps<RootStackParamList, 'Game'>;

Code: Using Typed Props in a Screen

Now, let's use these types in our GameScreen for full type safety and editor autocompletion.

// In screens/GameScreen.tsx
import { GameScreenProps } from '../App'; // Import the type

// Use the new type for our component's props
function GameScreen({ route, navigation }: GameScreenProps) {
  // SUCCESS: TypeScript now KNOWS route.params has a 'userNumber' of type 'number'!
  const { userNumber } = route.params;

  // ERROR: If you tried `route.params.ussrNumber`, TypeScript would give an error!
  // ...
}

Part 2: Componentization

Writing Cleaner, Reusable Code

What is Componentization?

Componentization is the process of breaking down a large user interface into smaller, independent, and reusable pieces called components.

Before (One big file)

<View style={card}>
  <Text>Title</Text>
  <TextInput />
  <View>
    <Pressable style={btn1}>...</Pressable>
    <Pressable style={btn2}>...</Pressable>
  </View>
</View>

After (Using Components)

<Card>
  <TitleText>Title</TitleText>
  <NumberInput />
  <PrimaryButton>...</PrimaryButton>
  <PrimaryButton>...</PrimaryButton>
</Card>

The "After" version is much easier to read, debug, and maintain.

The DRY Principle: Don't Repeat Yourself

Writing the same styling for every button or every card is tedious and error-prone. If you need to change the style, you have to find and edit every single instance.

Componentization helps us by:

  • Reusability: Build a <PrimaryButton> once, use it everywhere.
  • Maintainability: Need to change the button color? Edit one file: PrimaryButton.tsx.
  • Readability: <Card>...</Card> is much clearer than a <View> with 10 different style properties.

Refactoring Step 1: Creating Card.tsx

Our StartGameScreen uses a container with shadows and rounded corners. Let's turn this into a Card component.

// In components/Card.tsx
import { View, StyleSheet } from 'react-native';

function Card({ children }) {
  return <View style={styles.card}>{children}</View>;
}

export default Card;

const styles = StyleSheet.create({
  card: {
    padding: 16,
    marginTop: 36,
    marginHorizontal: 24,
    backgroundColor: '#3b021f', // A theme color
    borderRadius: 8,
    elevation: 4, // Android shadow
    shadowColor: 'black', // iOS shadow props
    shadowOffset: { width: 0, height: 2 },
    shadowRadius: 6,
    shadowOpacity: 0.25,
  },
});

Refactoring Step 2: Creating PrimaryButton.tsx

We will have many buttons in our app. Let's create a standard, reusable button component.

// In components/PrimaryButton.tsx
import { View, Text, Pressable, StyleSheet } from 'react-native';

function PrimaryButton({ children, onPress }) {
  return (
    <View style={styles.buttonOuterContainer}>
      <Pressable 
        style={styles.buttonInnerContainer} 
        onPress={onPress} 
        android_ripple={{ color: '#640233' }}
      >
        <Text style={styles.buttonText}>{children}</Text>
      </Pressable>
    </View>
  );
}

export default PrimaryButton;

// ... add styles for containers and text ...
